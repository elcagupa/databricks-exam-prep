Architecture — control plane vs data plane

Databricks splits responsibilities between two planes. The control plane is fully managed by Databricks and hosts everything the user interacts with — the web UI, REST API, job scheduler, cluster manager, and Unity Catalog metastore. The data plane lives inside the customer's cloud account and is where computation actually happens — driver nodes, worker nodes, DBFS, and external data sources. Critically, customer data never leaves the data plane.

ComponentPlaneManaged by

REST API / CLIControlDatabricks
Job schedulerControlDatabricks
Cluster managerControlDatabricks
UC metastoreControlDatabricks
Driver nodeDataCustomer cloud
Worker nodesDataCustomer cloud
DBFS rootDataCustomer cloud
JDBC data sourcesDataCustomer cloud
  • Driver node — orchestrates Spark tasks; hosts SparkContext; runs driver program
  • Worker nodes — execute tasks assigned by driver; hold executor JVMs
  • Executors — JVM processes on workers; hold partition data and run tasks
  • Serverless shift — in serverless mode, Databricks manages compute in their cloud account, shifting the data plane boundary
  • Secure cluster connectivity — outbound-only; no open inbound ports required in customer VNet/VPC
Exam trap: The web application is control plane. The driver node is data plane. These two are the most common distractors — don't swap them.

Query performance optimisation high frequency

Delta Lake provides multiple layers of performance optimisation. Each technique targets a specific bottleneck — file size, co-location, or point-lookup speed. Understanding when and how to apply each is heavily tested.

-- Compact small files into larger ones OPTIMIZE catalog.schema.my_table; -- Compact + co-locate data on filter columns OPTIMIZE my_table ZORDER BY (country, event_date); -- Liquid Clustering: modern replacement for Z-order -- On new table: CREATE TABLE events (id STRING, country STRING, dt DATE) CLUSTER BY (country, dt); -- On existing table: ALTER TABLE events CLUSTER BY (country, dt); -- Remove old data files permanently (default: 7 days) VACUUM my_table; VACUUM my_table RETAIN 168 HOURS; VACUUM my_table RETAIN 0 HOURS DRY RUN; -- preview only -- Bloom filter index for fast point lookups CREATE BLOOMFILTER INDEX ON TABLE events FOR COLUMNS(user_id OPTIONS (fpp=0.1)); -- Cache table in executor memory CACHE TABLE my_table; UNCACHE TABLE my_table; -- Update column statistics for query planning ANALYZE TABLE my_table COMPUTE STATISTICS FOR COLUMNS country, event_date; -- AQE config (auto-enabled DBR 7.3+) spark.conf.set("spark.sql.adaptive.enabled", "true") spark.conf.set("spark.sql.adaptive.skewJoin.enabled", "true") -- Broadcast join threshold (default 10MB) spark.conf.set("spark.sql.autoBroadcastJoinThreshold", "100m")

Liquid Clustering vs Z-order: Liquid Clustering is the modern recommendation. It handles multi-column clustering incrementally without a full OPTIMIZE run. Z-order still works but Liquid Clustering is preferred for new tables.

Spark internals

Understanding Spark's execution model helps interpret Spark UI output and answer optimisation questions.

  • SparkSession — unified entry point for DataFrame and SQL APIs
  • DAG — directed acyclic graph of transformations; built lazily until an action fires
  • Job — triggered by an action (collect, write, show, count)
  • Stage — set of tasks between shuffle boundaries; stages run sequentially
  • Task — one unit of work on one partition on one executor
  • Partition — chunk of data (~128MB default); one task processes one partition
  • Shuffle — redistribute data across executors between stages; most expensive operation
  • Catalyst optimizer — rewrites logical plan into an optimised physical plan
  • Lazy evaluation — transformations (filter, select, join) do nothing until an action is called

Lakehouse platform value

  • Delta Lake provides: ACID transactions, time travel, schema enforcement, Change Data Feed, unified batch + streaming
  • Unity Catalog provides: fine-grained access control, column lineage, auditing, Delta Sharing, Lakehouse Federation
  • Lakeflow family: Lakeflow Connect (ingest) → Lakeflow Declarative Pipelines (transform) → Lakeflow Jobs (orchestrate)
  • Open format — Parquet + Delta log = no vendor lock-in; readable by Spark, Trino, Athena, and more
  • Single platform — SQL, Python, Scala, R, ML, and streaming in one place
Exam trap: "Real-time notebook collaboration" is a workspace feature. "Support for both batch and streaming workloads" is a Delta Lake benefit. Don't mix these up.

Databricks Runtime (DBR)

  • Standard DBR — Spark + Python/Scala/R + Delta Lake; the baseline
  • ML Runtime — adds MLflow, scikit-learn, TensorFlow, PyTorch pre-installed
  • Photon Runtime — adds vectorised C++ engine on top of standard DBR
  • LTS (Long-Term Support) — supported 2+ years; use for stable production workloads
  • Databricks Connect requires DBR 13.0+ on the remote cluster
  • Cluster policies can enforce a specific DBR version across a team

Auto Loader high frequency

Auto Loader is the recommended way to continuously ingest new files from cloud storage into Delta Lake. It uses the cloudFiles format and supports two detection modes: directory listing (default — polls the folder periodically) and file notification (event-driven via SNS/SQS on AWS or Event Grid on Azure — more scalable for millions of files). Auto Loader guarantees exactly-once processing via checkpointing and handles schema evolution automatically.

# Complete Auto Loader read + write df = (spark.readStream .format("cloudFiles") # required .option("cloudFiles.format", "json") .option("cloudFiles.schemaLocation", "/chk/schema") # schema evolution .option("mergeSchema", "true") .option("cloudFiles.schemaEvolutionMode", "addNewColumns") .option("rescuedDataColumn", "_rescued") # performance throttling .option("cloudFiles.maxFilesPerTrigger", 1000) .option("cloudFiles.maxBytesPerTrigger", "10g") # file notification mode (high volume) .option("cloudFiles.useNotifications", "true") .load("/landing/events/") ) (df.writeStream .option("checkpointLocation", "/chk/events") .trigger(availableNow=True) # batch-like; preferred # .trigger(processingTime='5 minutes') # .trigger(once=True) # deprecated .outputMode("append") .option("mergeSchema", "true") .table("catalog.bronze.events") )
OptionValues / notes

mergeSchematrue = add new cols on schema change; false = strict enforcement (default)

Valid sources: Amazon S3, Azure Data Lake Storage Gen2, Google Cloud Storage. NOT valid for Auto Loader: local filesystem, JDBC, Kafka (use Spark Structured Streaming directly for those).

Auto Loader vs COPY INTO

Both tools ingest files idempotently, but they target different scenarios. Auto Loader is a streaming API for continuous high-volume ingestion. COPY INTO is a SQL command for ad-hoc or one-time batch loads.

Structured Streaming

Structured Streaming treats an incoming data stream as an unbounded table. You write the same DataFrame transformations you would for batch, and Spark handles the incremental execution. Checkpointing ensures exactly-once semantics and fault tolerance.

# Read from Delta table as stream stream_df = (spark.readStream .format("delta") .option("maxFilesPerTrigger", 10) .table("bronze.events") ) # Watermark + windowed aggregation windowed = (stream_df .withWatermark("event_time", "10 minutes") .groupBy( window(col("event_time"), "5 minutes"), col("country") ) .agg(count("*").alias("events")) ) # foreachBatch: custom write logic per micro-batch def process_batch(batch_df, batch_id): batch_df.write.mode("append").saveAsTable("silver.events") (stream_df.writeStream .foreachBatch(process_batch) .option("checkpointLocation", "/chk") .trigger(availableNow=True) .start() )

foreachBatch gives you a regular DataFrame per micro-batch — lets you write to multiple targets, apply MERGE, or use non-streaming sinks. Use it when the streaming writer alone isn't flexible enough.

Change Data Feed (CDF)

CDF tracks row-level changes (inserts, updates, deletes) in a Delta table, making it easy to propagate changes downstream — ideal for the Silver → Gold CDC pattern.

-- Enable CDF ALTER TABLE my_table SET TBLPROPERTIES ('delta.enableChangeDataFeed'='true'); -- Read changes between versions 1 and 5 SELECT * FROM table_changes('my_table', 1, 5); -- extra cols: _change_type, _commit_version, _commit_timestamp -- _change_type values: -- insert | delete | update_preimage | update_postimage # Read CDF as stream spark.readStream .option("readChangeFeed", "true") .option("startingVersion", 0) .table("my_table")

Medallion architecture high frequency

The Medallion Architecture organises data into three progressively refined layers. Each layer has a specific purpose and the exam tests your ability to identify which layer a given operation belongs to, and what the correct compute/pattern is for each.

Bronze — raw ingestion
  • Exact copy of source; no transformation
  • Append-only; full history preserved
  • Ingest with Auto Loader or COPY INTO
  • Add metadata cols: ingest_time, source_file
  • Schema: flexible — use rescue column
  • Retain all data; never purge early
Silver — cleaned & conformed
  • Filter nulls, remove duplicates
  • Enforce schema and data types
  • MERGE for CDC upserts
  • Join with dimension / lookup tables
  • Apply business rules and data quality
  • Enable CDF for Gold propagation
Gold — aggregated & curated
  • Business-level aggregations and KPIs
  • Optimised for BI and reporting
  • Wide denormalised tables
  • Use OPTIMIZE + Liquid Clustering
  • SQL Warehouse + Photon ideal here
  • Powers dashboards and ML features

Lakeflow Declarative Pipelines (formerly Delta Live Tables) renamed high frequency

Lakeflow Declarative Pipelines lets you declare Delta tables using SQL or Python — Databricks figures out execution order, manages retries, tracks data quality, and maintains lineage automatically. The LIVE. prefix references tables within the same pipeline.

-- SQL: streaming live table (incremental ingest) CREATE OR REFRESH STREAMING LIVE TABLE bronze_raw COMMENT "Raw events from landing zone" TBLPROPERTIES ("quality"="bronze") AS SELECT _metadata.file_name AS source_file, current_timestamp() AS ingest_time, * FROM cloud_files('/landing/', 'json', map('cloudFiles.schemaLocation', '/chk/schema')); -- SQL: live table with data quality expectations CREATE OR REFRESH LIVE TABLE silver_orders ( CONSTRAINT valid_amount EXPECT (amount > 0) ON VIOLATION DROP ROW, CONSTRAINT has_customer EXPECT (customer_id IS NOT NULL) ON VIOLATION FAIL UPDATE ) AS SELECT order_id, customer_id, amount, order_date FROM LIVE.bronze_raw WHERE order_date >= '2024-01-01'; -- SQL: gold aggregation table CREATE OR REFRESH LIVE TABLE gold_daily_revenue AS SELECT order_date, SUM(amount) AS total_revenue, COUNT(DISTINCT customer_id) AS unique_customers FROM LIVE.silver_orders GROUP BY order_date;
# Python equivalent import dlt from pyspark.sql.functions import * @dlt.table(comment="Raw events", table_properties={"quality":"bronze"}) def bronze_raw(): return (spark.readStream .format("cloudFiles") .option("cloudFiles.format", "json") .option("cloudFiles.schemaLocation","/chk") .load("/landing/")) @dlt.expect_or_drop("valid_amount", "amount > 0") @dlt.expect_or_fail("has_customer", "customer_id IS NOT NULL") @dlt.table() def silver_orders(): return dlt.read_stream("bronze_raw") # Read static table inside a pipeline @dlt.table() def enriched(): orders = dlt.read("silver_orders") customers = dlt.read("dim_customers") return orders.join(customers, "customer_id")

Delta Lake DDL / DML — complete reference high frequency

-- Create variants CREATE OR REPLACE TABLE t ( id STRING, dt DATE, v DOUBLE ) USING DELTA PARTITIONED BY (dt) CLUSTER BY (id) -- Liquid Clustering LOCATION 'abfss://...' -- external table COMMENT 'My table' TBLPROPERTIES ('delta.enableChangeDataFeed'='true'); CREATE TABLE IF NOT EXISTS t (...); -- skip if exists CREATE TABLE t2 AS SELECT * FROM t; -- CTAS -- Alter ALTER TABLE t ADD COLUMNS (new_col STRING); ALTER TABLE t ALTER COLUMN v COMMENT 'Revenue'; ALTER TABLE t RENAME TO t_new; ALTER TABLE t CLUSTER BY (id, dt); ALTER TABLE t SET TBLPROPERTIES('key'='val'); ALTER TABLE t ADD CONSTRAINT chk CHECK (amount > 0); -- enforced on write ALTER TABLE t DROP CONSTRAINT chk; -- DML INSERT INTO t VALUES ('a1','2024-01-01',100.0); INSERT INTO t SELECT * FROM staging; INSERT OVERWRITE t SELECT * FROM staging; UPDATE t SET v = v*1.1 WHERE dt='2024-01-01'; DELETE FROM t WHERE dt < '2023-01-01';
-- MERGE: upsert + CDC delete MERGE INTO silver.orders t USING bronze.raw s ON t.order_id = s.order_id WHEN MATCHED AND s.op='DELETE' THEN DELETE WHEN MATCHED AND s.updated_at > t.updated_at THEN UPDATE SET t.amount=s.amount, t.status=s.status WHEN NOT MATCHED THEN INSERT * WHEN NOT MATCHED BY SOURCE THEN DELETE; -- orphan cleanup -- Time travel SELECT * FROM t VERSION AS OF 5; SELECT * FROM t@v5; -- shorthand SELECT * FROM t TIMESTAMP AS OF '2024-06-01'; RESTORE TABLE t TO VERSION AS OF 3; RESTORE TABLE t TO TIMESTAMP AS OF '2024-01-01'; -- Maintenance OPTIMIZE t; OPTIMIZE t ZORDER BY (country, dt); VACUUM t RETAIN 168 HOURS; -- Inspect DESCRIBE TABLE t; -- columns + types DESCRIBE TABLE EXTENDED t; -- + location, properties DESCRIBE DETAIL t; -- file stats, size, format DESCRIBE HISTORY t; -- all versions + operations DESCRIBE HISTORY t LIMIT 5; -- last N versions DROP TABLE IF EXISTS t; DROP TABLE IF EXISTS t PURGE;-- also removes Delta log
Exam trap: CREATE OR REPLACE TABLE always creates (drops and recreates if exists). CREATE TABLE IF NOT EXISTS only creates if absent. INSERT OVERWRITE replaces all data but preserves table definition. These three are common wrong-answer traps.

Workflows — task types, repair, and orchestration high frequency

Databricks Workflows is the native orchestrator for multi-task jobs. Tasks are connected in a DAG via depends_on. The Repair & Rerun feature re-executes only failed tasks and their downstream dependents — a heavily tested concept.

# Pass values between tasks # In upstream task: dbutils.jobs.taskValues.set(key="row_count", value=12345) # In downstream task: count = dbutils.jobs.taskValues.get( taskKey="ingest_bronze", key="row_count", default=0 )
  • Repair & Rerun — re-runs the failed task and all downstream dependents only; already-succeeded tasks are skipped. This is always the correct answer for "efficient failure recovery" scenarios.
  • Retry policy — set max_retries and min_retry_interval_millis per task
  • Conditional run — run_if: ALL_SUCCESS / AT_LEAST_ONE_SUCCESS / NONE_FAILED / AT_LEAST_ONE_FAILED / ALL_DONE
  • Timeout — timeout_seconds per task or per job; auto-cancels overrunning jobs
  • Max concurrent runs — prevent scheduled runs from overlapping
  • File arrival trigger — polls a path for new files; starts job when detected
  • Task cluster options — job cluster (dedicated), existing all-purpose, serverless, or pool
Orchestration patterns: Fan-out = task A triggers [B, C, D] in parallel. Fan-in = [A, B] must both succeed before C starts. Diamond = fan-out + fan-in combined. Define the DAG using depends_on lists.
Exam trap: "Repair run" re-executes from the failed task — not from the beginning. Choosing "re-run the entire job" is the wrong answer for recovery scenarios.

Unity Catalog — hierarchy, roles, and permissions high frequency

Unity Catalog is the centralised governance layer for all Databricks assets. It uses a three-level namespace — catalog.schema.table — and a privilege model where users must have USE access at every level of the hierarchy before object-level privileges take effect. This hierarchical requirement is the most frequently tested concept in this domain.

-- Minimum read access (all three required) GRANT USE CATALOG ON CATALOG prod TO analysts; GRANT USE SCHEMA ON SCHEMA prod.sales TO analysts; GRANT SELECT ON TABLE prod.sales.orders TO analysts; -- Schema-level (applies to all objects in schema) GRANT SELECT ON SCHEMA prod.sales TO analysts; GRANT MODIFY ON TABLE prod.sales.orders TO `eng-team`; GRANT CREATE TABLE ON SCHEMA dev.staging TO `eng`; GRANT CREATE VIEW ON SCHEMA dev.staging TO `eng`; -- Catalog-level GRANT ALL PRIVILEGES ON CATALOG dev TO `dev-team`; GRANT CREATE SCHEMA ON CATALOG dev TO `eng`; -- Function privileges GRANT EXECUTE ON FUNCTION prod.utils.fn TO analysts; -- Revoke and inspect REVOKE SELECT ON TABLE t FROM analysts; SHOW GRANTS ON TABLE prod.sales.orders; SHOW GRANTS TO analysts; SHOW GRANTS ON CATALOG prod;

Managed vs external tables high frequency

The key distinction is who owns the data lifecycle. With managed tables, Unity Catalog controls both the metadata and the underlying data files — dropping the table deletes everything. With external tables, you own the files; dropping the table only removes the metadata entry in UC.

-- Managed: UC controls data + metadata lifecycle CREATE TABLE prod.sales.orders ( order_id STRING, amount DOUBLE, dt DATE ) USING DELTA; -- Data stored at UC-managed path automatically -- DROP TABLE = deletes data files + metadata -- External: you control the data location CREATE TABLE prod.sales.orders_ext ( order_id STRING, amount DOUBLE ) USING DELTA LOCATION 'abfss://container@storage.dfs.core.windows.net/orders'; -- DROP TABLE = removes UC metadata only -- Data files remain in ADLS/S3/GCS -- Check type DESCRIBE DETAIL prod.sales.orders; -- "type" column = MANAGED or EXTERNAL -- External location + storage credential (prerequisite) CREATE STORAGE CREDENTIAL my_cred WITH AZURE MANAGED IDENTITY APPLICATION_ID 'xxx'; CREATE EXTERNAL LOCATION my_loc URL 'abfss://container@storage.dfs.core.windows.net/' WITH (STORAGE CREDENTIAL my_cred); GRANT READ FILES ON EXTERNAL LOCATION my_loc TO `eng-team`;

Delta Sharing expanded topic high frequency

Delta Sharing is an open protocol for sharing live Delta data without copying or moving it. Recipients query your storage directly via signed URLs — data never leaves your account. Two flavours: D2D (recipient has Databricks) and D2O (recipient uses any platform via the open protocol).

-- Step 1: create a share CREATE SHARE orders_share COMMENT 'Orders data for partners'; -- Step 2: add tables to share ALTER SHARE orders_share ADD TABLE prod.sales.orders; ALTER SHARE orders_share ADD TABLE prod.sales.orders PARTITION BY (dt) -- filter by partition AS 'shared_orders'; -- alias for recipient ALTER SHARE orders_share ADD SCHEMA prod.sales; -- share entire schema -- Step 3: create recipient CREATE RECIPIENT acme_corp COMMENT 'External partner ACME'; -- Returns: activation_link (send to recipient) -- Step 4: grant recipient access GRANT SELECT ON SHARE orders_share TO RECIPIENT acme_corp; -- Inspect and revoke SHOW SHARES; DESCRIBE SHARE orders_share; SHOW RECIPIENTS; REVOKE SELECT ON SHARE orders_share FROM RECIPIENT acme_corp; DROP RECIPIENT acme_corp;

Lakehouse Federation new topic

Lakehouse Federation lets you query external databases directly from Databricks — without moving the data. The external source is registered as a foreign catalog in Unity Catalog and is governed with the same GRANT/REVOKE model as native tables.

-- Step 1: create a connection to external DB CREATE CONNECTION pg_prod TYPE postgresql OPTIONS ( host 'db.prod.example.com', port '5432', database 'orders_db', user 'readonly_user', password secret('my-scope', 'pg-password') ); -- Step 2: expose as foreign catalog in UC CREATE FOREIGN CATALOG pg_prod_catalog USING CONNECTION pg_prod; -- Step 3: query like any UC object SELECT * FROM pg_prod_catalog.public.orders LIMIT 100; -- Cross-system join (Databricks + external) SELECT o.order_id, c.name FROM pg_prod_catalog.public.orders o JOIN prod.sales.customers c ON o.cust_id = c.id; -- Grant access to foreign catalog GRANT USE CATALOG ON CATALOG pg_prod_catalog TO analysts; GRANT SELECT ON TABLE pg_prod_catalog.public.orders TO analysts;

Lineage, audit logs, and system tables

Unity Catalog automatically tracks column-level and table-level lineage — no setup required. All access events are logged to system tables, making them queryable just like any Delta table. This enables both governance audits and operational observability.

-- Audit: who accessed what and when SELECT event_time, user_name, action_name, request_params.table_full_name AS table_name FROM system.access.audit WHERE action_name = 'commandSubmit' AND event_date = current_date() ORDER BY event_time DESC; -- Table lineage SELECT * FROM system.access.table_lineage WHERE target_table_full_name = 'prod.sales.orders'; -- Column lineage SELECT * FROM system.access.column_lineage WHERE target_table_full_name = 'prod.sales.orders'; -- Billing usage (last 30 days) SELECT workspace_id, sku_name, SUM(usage_quantity) AS dbus FROM system.billing.usage WHERE usage_date >= dateadd(DAY,-30,current_date()) GROUP BY 1, 2;
-- Tag columns for PII tracking ALTER TABLE prod.sales.customers ALTER COLUMN email SET TAGS ('pii'='true', 'sensitivity'='high'); ALTER TABLE prod.sales.customers SET TAGS ('domain'='customer', 'owner'='data-eng'); SHOW TAGS ON TABLE prod.sales.customers; SHOW TAGS ON COLUMN prod.sales.customers.email; -- Row-level security via dynamic view CREATE VIEW secure_orders AS SELECT * FROM orders WHERE region = current_user() -- filter by logged-in user OR IS_MEMBER('admins'); -- admins see everything -- Data quality constraint (enforced on write) ALTER TABLE t ADD CONSTRAINT chk_amount CHECK (amount > 0); -- Secrets management pw = dbutils.secrets.get(scope="my-scope", key="db-pass") # Always prints as [REDACTED] in notebook output -- In SQL connection strings -- password secret('my-scope','db-password')
CHECK vs Lakeflow expectation: A CHECK constraint is enforced at write time — violations cause an error and the write fails. Lakeflow pipeline expectations offer three modes (warn, drop, fail) and are better suited for data quality monitoring with metrics tracking.
